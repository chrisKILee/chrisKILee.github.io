코드가 사양을 따르는 패러다임 전환을 통해 AI 코딩 어시스턴트(Claude, Cursor, Windsurf)를 최대로 활용하고 일관된 품질을 확보하세요.

프로세스 시작하기

기존 개발 vs SDD

GitHub Spec-Kit 방법론(2024년 9월)을 기반으로 한 SDD는 개발 과정을 역전시킵니다. 사양이 코드를 가이드하는 것이 아니라, 사양이 AI 해석을 통해 코드를 생성합니다.

  • 진실의 원천: 코드가 아닌 사양 문서가 중심
  • 인간의 역할: 코더에서 설계자 및 검토자로 전환
  • AI의 역할: 사양을 기반으로 구현체 생성 (TDD 준수)
Traditional SDD Paradigm
요구사항 해석 (인간) AI가 사양 해석
코드 작성 (인간) AI가 코드 생성
테스트 작성 (후순위) 구현 전 TDD 필수

4단계 SDD 워크플로우

각 단계마다 인간의 검토(Human Checkpoint)가 필수적입니다.

01

Specify (정의)

/speckit.specify

'무엇'과 '왜'를 캡처합니다. 사용자 여정, 비즈니스 요구사항, 성공 기준을 정의합니다.

기술적 결정(Stack 등)은 의도적으로 배제합니다.
02

Plan (계획)

/speckit.plan

요구사항을 기술적 설계로 변환합니다. 아키텍처 패턴, 기술 스택, 의존성을 선택합니다.

AI가 옵션을 조사하고 Constitution에 맞는 솔루션을 제안합니다.
03

Tasks (작업 분해)

/speckit.tasks

계획을 원자 단위의 병렬 처리 가능한 작업으로 분해합니다.

의존성이 없는 작업은 [P]로 표시하여 동시에 실행할 수 있습니다.
04

Implement (구현)

/speckit.implement

TDD 방식으로 작업을 실행합니다. 테스트가 구현보다 선행됩니다.

각 단계별 인간의 코드 리뷰 및 검증이 필수입니다.

Constitution.md (헌법)

프로젝트의 협상 불가능한 원칙을 정의하는 가장 중요한 파일입니다. AI의 모든 결정은 이 파일을 따릅니다.

핵심 원칙 (Core Principles)

  • DDD (Domain-Driven Design) 준수
  • TypeScript Strict Mode 필수
  • 모든 공개 API는 OpenAPI 사양 필수
  • 구조화된 로깅을 통한 에러 처리

품질 게이트 (Quality Gates)

  • 테스트 커버리지 최소 80%
  • 치명적 보안 취약점 0건
  • API 응답 시간 < 200ms (p95)

디렉토리 구조 & 규칙 

.specify/              # Spec-Kit 설정
memory/                # AI 메모리
constitution.md        # [중요] 프로젝트 헌법
specs/                 # 사양 저장소
  ├── 001-feature-name/
  │    ├── spec.md     # 기능 명세서
  │    ├── plan.md     # 기술 구현 계획
  │    ├── tasks.md    # 작업 목록
  │    └── research.md # 기술 조사 내용
contracts/
  └── api-spec.json    # API 계약
templates/             # 템플릿
CLAUDE.md              # Claude 컨텍스트 규칙
.cursor/rules/         # Cursor 규칙 (*.mdc)

규칙 적용 우선순위

로컬(~/.claude) < 프로젝트 루트 < 백엔드/프론트엔드 폴더별 규칙
하위 폴더의 구체적인 규칙이 상위 규칙보다 우선합니다.

기술 스택 및 품질 보증 전략

Enterprise Stack

  • Nx Monorepo (Scope + Type 분류)
  • Frontend: React 18+, TypeScript
  • Backend: NestJS (DDD 구조)
  • DB: PostgreSQL + Prisma ORM
  • Infra: Kubernetes + ArgoCD

Test Pyramid

  • Unit (60-70%): 비즈니스 로직
  • Integration (20-25%): 모듈 간 연동
  • Contract (10-15%): Pact 활용
  • E2E (5-10%): 전체 흐름

Success Metrics

  • Spec 완료율 > 90% (코딩 전)
  • Spec-Code 일치율 > 95%
  • AI 구현 성공률(초안) > 70%
  • 재작업률(Rework Rate) < 10%

90일 도입 로드맵

Phase 1
Day 1-30

Pilot (파일럿)

핵심 팀 3-5명. 헌법 수립, 도구 세팅, 챔피언 육성.

Phase 2
Day 31-60

Expansion (확장)

추가 5-10명. 챔피언과의 페어 프로그래밍. 새로운 기능의 50%에 SDD 적용.

Phase 3
Day 61-90

Org-Wide (전사 도입)

모든 엔지니어링 팀. 기능의 80% 이상 SDD 적용. DORA 지표 개선 확인.
